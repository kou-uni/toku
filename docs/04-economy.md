# 04. TOKU トークン経済

> このドキュメントは、TOKU の発行・流通・寄付の経済設計を、なぜそうなっているかを含めて記述します。
> 投機トークンではなく、**反投機の文化トークン**として設計されている点が要です。

---

## 設計の根本原理

> **持つと減り、渡すと満ちる。**
> **投機の逆向き、回向の経済。**

通常のトークンエコノミー:
> 発行量 ↑ → 1単位の価値 ↓ (インフレ)

TOKU:
> 発行量 ↑ → 灯火プール ↑ → DAO 寄付 ↑ → TOKU の文化的重み ↑ (反インフレ)

これが**ポイ活ではなく、自分を支える内面の構造**たる所以である。

## TOKU の発行(Mint)

### 発行源と量

| 行為 | 発行量 | 説明 |
|---|---|---|
| 3分以上の座 | 1 TOKU | 標準セッション完了 |
| 5分以上の座 | 1.5 TOKU | 5分セッション |
| 10分以上の座 | 3 TOKU | 10分セッション |
| 灯火プールへの投稿 | 0.5 TOKU | reflection を匿名で流す |
| 灯火が誰かに届いた | 1.5 TOKU(後追い) | 自分の灯火が draw された |
| 未来の種の開封 | 2 TOKU | 7日後 / 30日後 等の TimeLock |
| ペイフォワード送付 | 1 TOKU(送る側に) | 構造的に渡す動機 |
| 受領(初回) | 1 TOKU | gift link を受け取った時 |

### 重要な原則
- **発行は記録に基づく**(セッションは Sui の object として証跡が残る)
- **管理者裁量での発行は無し**(コードで自動)
- **Mint は always with attestation**(発行理由が常に object に紐づく)

## Decay(徳の薄れ)

### 仕組み
- TOKU は持っているだけでは **月 1% ずつ減少** する
- ただし以下のいずれかをすれば、**減少が一時停止 + 直近の減少分が回復**:
  - 新しいセッションを行う
  - 灯火プールに投稿する
  - 誰かにペイフォワード送付する
  - 灯火を購入する

### Decay の哲学的根拠

仏教における「**諸行無常**」と「**回向**」:
- 持ち続けることに意味がない
- 流すことで初めて、徳の本質が現れる

行動経済学的:
- **損失回避** を pro-社会的方向に転用
- 「貯めると損」が「渡すと得」に反転

### 実装(`toku.move`)

```move
public struct TokuBalance has key {
    id: UID,
    amount: u64,
    last_active_at: u64,
    last_decay_applied_at: u64,
}

public fun apply_decay(balance: &mut TokuBalance, clock: &Clock) {
    let now = clock::timestamp_ms(clock);
    let inactive_days = (now - balance.last_active_at) / (24 * 60 * 60 * 1000);
    if (inactive_days > 30) {
        let decay_rate = 100; // 1% per month, expressed as basis points
        let decay_amount = (balance.amount * decay_rate) / 10000;
        balance.amount = balance.amount - decay_amount;
        balance.last_decay_applied_at = now;
    }
}
```

## Pay-it-forward の数学

### 基本ボーナス
```
送る側:  +1 TOKU(送付ボーナス)
受け手:  +1 TOKU(受領ボーナス、初回のみ)
合計:    送った量 + 2 TOKU(エコシステムが純増)
```

### なぜ純増を許すのか
- 純増分は **新規ユーザー獲得のコスト**
- 新規ユーザーは、四半期 Tide で寄付プールに貢献する潜在
- つまり、**徳の発行 = 公益の先取り**

### 連鎖ボーナス(将来)
- 受け取った人がさらに別の人に送ると、最初の人にも追加 TOKU
- 縁起ボーナス、3段までは効く設計
- ネットワーク効果を構造化

## 灯火プール経済(決済モックアップ)

### 流れ
```
[投稿側]
   reflection を匿名で投稿
        ↓
   LanternCard が pool に追加
        ↓
   投稿者に +0.5 TOKU
        ↓
   ───── ここで一時停止 ─────

[購入側]
   別ユーザーが「灯火がほしい」とリクエスト
        ↓
   Suiネイティブ決済モック: /api/lantern へリクエストし、ダミーレスポンスを受領
        ↓
   (将来構想: 0.05 USDC相当のSUIが Tide Pool に流入)
        ↓
   ランダムに 1 枚の LanternCard が選択
        ↓
   購入者に表示、投稿者に +1.5 TOKU(後追い)
```

### 価格設定
- **将来構想**: 非投機的で手軽な価格設定
- 投稿者と購入者の双方に正のインセンティブ
- 将来の決済流入額の 80% は Tide Pool へ(寄付の原資)
- 20% はインフラ運用費

### 将来のSui連携
本番(将来)では Sui のネイティブ決済機能と組み合わせるが、現MVPでは:
- ダミーのモックアップAPIを実装
- アーキテクチャの複雑化を避けるため、実際の決済処理は割愛

## Tsumu Tide(寄付プール)

### 仕組み
- 将来の決済流入額の 80% が Tide Pool に蓄積
- 四半期に1回、TOKU holder で**寄付先投票**
- 結果は Sui に永続記録、誰でも参照可能

### 投票権
- 1 TOKU = 1 票(連続)
- ただし **Decay により最近積んだ人の発言力が大きい**
- 過去 90日間に積んだ TOKU のみが投票に効く(設計案)

### 寄付候補(初期)
1. 全国子ども食堂支援センター(認定NPO)
2. 心のサポートライン(自殺予防 NPO)
3. 山寺・禅道場の維持(京都府宗教法人協議会経由)
4. 心理学・脳科学の研究機関(京都大学・東京大学)

### 透明性
- すべての proposal、vote、distribution は Sui object として記録
- 誰でも `https://suiscan.xyz/...` で確認可能
- 寄付先の受領証も object に紐づく

## 反投機性の保証

### 設計上、TOKU は以下を満たさない
- DEX で売買できる(送付は claim link 経由のみ、転送先は人間)
- 投機トレーダーが価値を吊り上げる(Decay により貯めても減る)
- whale が支配する(投票は最近積んだ TOKU のみ有効)

### 設計上、TOKU は以下を満たす
- 自分の実践の証跡である(Session object と紐づく)
- 他者を支える原資になる(灯火経由)
- 社会の善の一部に変換される(Tide DAO)
- 永遠に所有できる(Sui object として記録)

## なぜ「経済設計」の話を真剣にするのか

トークン経済の設計を間違えると、Tsumu は単なる **「徳ガチャ」** になる。
ガチャは続くが、人を救わない。

設計の核心:

> **トークンの動きが、ユーザーの内面の動きと、構造的に一致している**

- 内面で「持ちすぎると重くなる」 ⇄ Decay
- 内面で「渡すと軽くなる」 ⇄ Pay-forward bonus
- 内面で「自分の言葉が誰かに届く嬉しさ」 ⇄ 灯火還流
- 内面で「自分が小さくても社会に貢献している」 ⇄ Tide DAO

これが反転した時、ユーザーは Tsumu を **「習慣アプリ」** ではなく **「自分を支える装置」** として理解する。

そしてそれは、**ハッカソン審査員にも同じく刺さる**。
